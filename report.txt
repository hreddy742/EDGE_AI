## 1. EXECUTIVE SUMMARY

This report is a complete engineering-grade audit of 17 GitHub repositories from the "Retail Theft / Shoplifting Detection Research Sources" document, followed by a deep file-level audit of the EdgeGuard (EDGE_AI) codebase, a cross-repo comparison, and a final upgrade blueprint.

Key findings in brief:

- Of the 17 external repos, only 3 are genuinely production-relevant (Repos 1, 4, 11). The rest are student notebooks, hackathon demos, or non-functional scaffolds.
- EdgeGuard is architecturally the most production-complete system in this entire review — it has temporal FSM logic, zone awareness, multi-camera ReID, evidence clips, a FastAPI backend, Streamlit UI, and SQLite persistence that no other repo comes close to.
- EdgeGuard's critical gap is that its detection brain remains single-model (YOLOv8n only), with no training pipeline, no benchmark/evaluation tooling, no swappable model framework, no advanced concealment models, and no temporal video classifier.
- The most valuable borrowable ideas from the external repos are: optical flow-based temporal modeling (Repo 4), POS/counter transaction correlation (Repo 3), pose+XGBoost hybrid approach (Repo 13), GCAE+Transformer anomaly detection (Repo 7), and full-stack alert delivery (Repo 1).

### REPO 1 AUDIT — vahapogut/Theft-Detection ("TheftGuard AI")

1. REPO OVERVIEW
Solves the problem of real-time shoplifting detection in retail environments with a complete full-stack system.
Covers surveillance, behavioral analysis, face recognition, multi-camera management, alerting, and historical analysis.
The most production-complete external repo reviewed. Production-grade architecture, though with some rough edges.

2. PROJECT STRUCTURE
  backend.py          FastAPI server — all detection, WebSocket, camera management
  main.py             Standalone/demo detection pipeline
  settings.json       Email, Telegram, ROI config
  dashboard/          Next.js 14 frontend
  yolov8n.pt          Object detection weights
  yolov8n-pose.pt     Pose estimation weights
  shoplifting.pt      Optional custom theft classifier
  theft_detection.db  SQLite
  alerts/             Snapshot storage

Entry point: backend.py (FastAPI app). No requirements version pinning.

3. CODE ARCHITECTURE
Monolithic backend.py handles everything: camera thread management, YOLO inference, pose estimation, face
recognition, ROI polygon checks, alert delivery, and WebSocket streaming. Not modular — everything is in one
file. The Next.js frontend is decoupled and communicates via REST + WebSocket.

4. MODEL AUDIT

  Model               | Purpose                               | Lightweight? | Real-time? | Edge viable?
  YOLOv8n             | Person + 17 high-theft-risk classes   | Yes (nano)   | Yes        | Yes (CPU viable)
  YOLOv8n-pose        | Wrist/shoulder keypoints              | Yes          | Yes        | Yes
  shoplifting.pt      | Custom theft action classifier        | Unknown      | Unknown    | Unknown (black box)
  face_recognition    | Blacklist/VIP identification          | Medium       | Medium     | No (dlib heavy)

Object detection runs every 5 frames. Pose runs every frame.

5. FEATURE AUDIT
- Person detection: YOLOv8n, every 5th frame
- Pose estimation: YOLOv8n-pose, per-frame, wrist and shoulder keypoints
- ROI zones: User-drawn polygons via mouse callback; OpenCV pointPolygonTest() for inside/outside
- Behavioral heuristics:
    * Hand-to-torso distance below threshold -> concealment flag
    * Shoulder-to-hip vertical anomaly -> bending detection
    * 5-second dwell time -> loitering alert
    * Alert cooldown: 3 seconds per camera
- Face recognition: Every 2 seconds per tracked person; compares against SQLite-stored encodings (pickled numpy);
  supports blacklist + VIP whitelist
- Multi-camera: Multiple RTSP/webcam sources managed via concurrent threads
- Alerting: Email via SMTP (image attachment) + Telegram Bot API (photo + caption)
- WebSocket: {type, cameras[{camera_id, name, base64_jpg}], alert, audio} at 25fps
- Dashboard: Next.js 14; historical alert table, statistics, export
- Database: SQLite — alerts (UUID, message, timestamp, image) + faces (encoding, type)
- Custom model path: Falls back gracefully from shoplifting.pt to heuristics if not present

6. LOGIC AUDIT
Detection logic — three-tier priority:
  1. Custom model: if shoplifting.pt present -> classify frames, fire on "shoplift"/"suspicious"/"theft" label
  2. Heuristics: wrist-to-torso distance, bending, loitering, zone intrusion (all simple Euclidean thresholds)
  3. Face match: immediate alert on blacklisted face regardless of behavior

Theft decision: rule-based (thresholds) with optional ML override. No temporal sequence logic. No suspicion
score accumulation. Single-trigger model — one frame that trips the threshold generates an alert.

Zone logic: Polygonal ROIs drawn by user at runtime. Person center point tested against polygon. No shelf/exit
distinction — single generic "suspicious zone."

False positive suppression: 3-second cooldown between alerts per camera. Insufficient for production.

7. DATA AUDIT
Uses COCO-pretrained YOLOv8n. No custom dataset included. shoplifting.pt is a black box — training data and
methodology unknown. Face encodings stored per-user as pickled numpy arrays in SQLite.

8. TRAINING AUDIT
No training code in repo. shoplifting.pt was trained externally — details unavailable.

9. INFERENCE AUDIT
- Input: RTSP streams or webcam (OpenCV VideoCapture)
- Output: annotated WebSocket frames + SQLite events + file snapshots
- FPS: ~25fps WebSocket delivery; inference throttled to every 5th frame
- Multi-camera: yes, concurrent threads
- CPU viable: yes for detection; face_recognition requires more CPU
- Edge deployment: partially viable — face recognition (dlib) problematic on embedded hardware

10. PERFORMANCE / PRODUCTION
Would work in controlled environments. 24/7 stability uncertain — no error recovery on RTSP dropout documented.
Multi-camera scaling limited by threading model. The 3-second cooldown is too short for real retail.

11. CODE QUALITY
Monolithic backend.py is a weakness. No tests. No version pinning. No Docker. Next.js dashboard is separate
and well-structured. SQLite is not production-grade for multi-camera high-throughput logging.

12. REUSABLE INSIGHTS FOR EDGE_AI
- Face recognition blacklist concept — valuable for known-offender flagging
- Email + Telegram dual-channel alerting pattern
- Next.js dashboard with historical export functionality
- Graceful model fallback (primary model -> heuristics)

13. FINAL VERDICT
Practical: Yes (most complete external repo). Edge-friendly: Partially (face recognition problematic).
Useful: Yes for dashboard/alerting patterns. Best parts: multi-channel alerts, face recognition, Next.js UI.
Biggest weakness: monolithic code, black-box custom model, no temporal logic.

### REPO 2 AUDIT — OmRajpurkar/Alert-Generation (Transfer Learning)

1. REPO OVERVIEW
IEEE ICCCNT 2020 paper implementation. Solves multi-class suspicious activity classification (Shoplifting,
Robbery, Break-In) using ResNet50 transfer learning. Research-grade with published methodology but low
production value.

2. PROJECT STRUCTURE
  videoClassification/
    train.py                    ResNet50 training (3-class)
    predict_video.py            Batch video inference
    predict_video_realtime.py   Webcam real-time inference
    model                       Saved weights
    label_bin                   Pickled LabelBinarizer

3. CODE ARCHITECTURE
Clean three-file separation: train / batch inference / real-time inference. No API. No UI. No database.

4. MODEL AUDIT
Model: ResNet50 (frozen) -> AveragePool2D(7x7) -> Flatten -> Dense(512, ReLU) -> Dropout(0.5) -> Dense(3, Softmax)
Input: 224x224 BGR->RGB with ImageNet mean subtraction [123.68, 116.779, 103.939]
Lightweight: Medium. Real-time: Yes (on GPU). Edge viable: Borderline (200MB+)

5. FEATURE AUDIT
- Temporal aggregation: Rolling deque of 128 frame predictions, averaged
- Training pipeline: SGD optimizer, lr=1e-4, momentum=0.9; data augmentation (30 deg rotation, 0.15 zoom,
  shifts, horizontal flip); stratified 80/20 split; ROC curves + confusion matrix
- Alert threshold: max class prob - sum of others > 0.80 (interesting multi-class decision rule)
- No tracking, no bounding boxes, no zones, no alerting integration

6. LOGIC AUDIT
Pure image classification applied frame-by-frame with rolling average smoothing. No spatial awareness.
No temporal sequence modeling beyond the averaging trick. Theft decision = probability threshold comparison.
Three-class decision: which class wins AND exceeds 0.80 margin.

7. DATA AUDIT
Private dataset (must download separately). Three subdirectories: Shoplifting/, Robbery/, Break-In/.
Only class labels, no bounding boxes. Data quality unknown.

8. TRAINING AUDIT
Clean and functional training script. SGD with momentum. Standard augmentation. Proper train/val split.
Outputs training curves.

9. INFERENCE AUDIT
- Frame-level: 224x224 resize + normalize -> predict -> deque
- FPS: fast (ResNet50 inference ~30-50fps on GPU)
- No multi-person support — single global frame classification

10. PRODUCTION
Not production-grade. No API, no alerting, no zones, no tracking. Research prototype only.

12. REUSABLE
- Multi-class (Shoplifting + Robbery + Break-In) approach worth borrowing conceptually
- Rolling 128-frame probability averaging is a simple and effective smoothing technique

13. VERDICT
Not practical for EDGE_AI use case. Research reference only. The 3-class crime taxonomy is a useful
conceptual idea.

### REPO 3 AUDIT — raoh97-code/theft-detection-system (POS Correlation)

1. REPO OVERVIEW
Hackathon project solving the retail drawer-theft problem via POS log correlation. Novel approach:
cross-reference video events (person presence, cash drawer movement) against point-of-sale transaction logs.

2. PROJECT STRUCTURE
  ai-engine/
    detect.py          YOLOv8 person tracking -> events.json
    motion.py          Cash drawer motion detection
    event_generator.py JSON event serialization
  backend/
    server.js          Node.js API
    anomalyDetector.js POS correlation engine
  frontend/
    LiveFeed.js, Alerts.js, Timeline.js
  pos-simulator/
    generate_logs.js   Fake POS log generator
  shared/
    events.json, alerts.json  Data exchange (flat files)

Languages: JavaScript 40.4% / CSS 30.6% / HTML 24.5% / Python 3.7% / PowerShell 0.8%

3. CODE ARCHITECTURE
Multi-layer: Python AI engine writes JSON -> Node.js backend reads + correlates -> React frontend displays.
Communication via flat JSON files (not a proper message bus). Python is only 3.7% of codebase.

4. MODEL AUDIT
  YOLOv8n          | Person tracking (entry/exit detection) | Only tracks presence, not behavior
  OpenCV motion    | Cash drawer movement                   | Simple frame differencing

5. FEATURE AUDIT
- Person entry/exit tracking via set operations on detection IDs
- Cash drawer motion detection (OpenCV)
- POS log correlation with 3-tier confidence scoring:
    * Drawer open + no POS transaction within 2s -> 90% confidence
    * Suspicious cash motion -> 85% confidence
    * Drawer open + no person -> 95% confidence (tamper)
- React dashboard with timeline view
- alerts.json output (no database)

6. LOGIC AUDIT
The most domain-realistic theft logic in all 17 repos. Addresses the actual problem: an employee or thief
opens a cash drawer without recording a sale. Cross-referencing video person presence with POS logs is how
real loss prevention works. The 2-second timestamp matching window is sensible.

Three explicit confidence levels represent risk scoring — more nuanced than binary classification.

7-8. DATA / TRAINING
No training. All rule-based. POS simulator generates fake logs for testing.

9. INFERENCE
FRAME_SKIP=5 for performance. Exports person entry/exit events to JSON. Node.js polls JSON every 2 seconds.
Not real-time streaming — polling-based.

10. PRODUCTION
Not production-ready. Flat JSON files instead of database. POS integration is simulated. No authentication.
Hackathon origin.

12. REUSABLE
- POS correlation concept is highly valuable — cross-referencing video events with transaction logs is a
  critical missing feature in EDGE_AI
- 3-tier confidence scoring model is worth borrowing
- Counter-level tamper detection (drawer open with no person) is novel

13. VERDICT
Weak implementation, but strongest conceptual insight of all 17 repos for realistic retail theft.
The POS correlation module should be a V2 priority for EDGE_AI.

### REPO 4 AUDIT — 1amitos1/Shoplifting-Detection (Gate_Flow_SlowFast)

1. REPO OVERVIEW
The most technically sophisticated ML system in all 17 repos. Implements a dual-stream 3D-CNN
(Gate_Flow_SlowFast) trained on 4,000 real supermarket clips to detect bag and clothing concealment.
Published F1: 85.77%.

2. PROJECT STRUCTURE
  CODE/
    Shoplifting_net.py          Gate_Flow_SlowFast model (TF/Keras)
    Live_demo_Shoplifting.py    Threaded real-time inference + alerts
    data_pip_shoplifting.py     Data pipeline, optical flow, augmentation
    Alert.py                    Email alert (hardcoded Gmail credentials — DO NOT COPY)
    weight_steals/              Pre-trained Bag/Clothes concealment weights
    weight_take_Somthing/       Pre-trained taking behavior weights
  DB_Sample/                    Sample dataset clips

3. CODE ARCHITECTURE
Three-thread architecture for real-time: Receive (capture) / Display (buffer) / Prediction (inference).
3000-frame queue buffer. Custom TF/Keras model. No API or web interface — standalone scripts only.

4. MODEL AUDIT
  Component        | Architecture                              | Purpose
  Fast pathway     | Conv3D (1,3,3)+(3,1,1), 16 filters       | Fine-grained temporal dynamics (all 64 frames)
  Slow pathway     | Conv3D, tau=16 temporal subsample         | Semantic spatial features (4 frames effectively)
  Lateral connect  | Addition at stride=18                     | Cross-pathway fusion
  Merge block      | Conv3D (64->128), pooling (2,2,2)         | Combined feature integration
  Head             | Dense(128->32->3, Softmax)                | Bag / Clothes / Normal classification

Input: [64, 224, 224, 5] — 64 frames, 224x224, RGB + 2-channel optical flow.
This is the ONLY repo using optical flow.

5. FEATURE AUDIT
- Optical flow computation: between consecutive frames, integrated as 2 additional channels
- 64-frame sliding window: ~2 seconds at 30fps, with 32-frame stride overlap
- Multi-window aggregation: max confidence across all windows for long videos
- Threaded real-time inference: separate capture, display, prediction threads
- Email alert with video attachment: Gmail SMTP, base64-encoded video file
- Three output classes: Bag, Clothes, Normal — more nuanced than binary

6. LOGIC AUDIT
Purely learned temporal pattern. No heuristics. The 3D-CNN learns to recognize concealment sequences
from training data. Multi-window approach: run model on overlapping 64-frame windows, take max confidence.
Alert fires when Bag or Clothes wins.

The optical flow addition is the key insight: [RGB(3) + flow(2)] = 5 channels. Flow vectors encode where
pixels moved, revealing hand trajectories during concealment even when RGB appearance is ambiguous.

7. DATA AUDIT
4,000 video clips from 2 real supermarkets. Actor-performed scenarios with realistic product/bag/clothing
concealment. Multiple angles, actors, clothing variants. 87% F1 on training, 85.77% on evaluation.
Dataset: contact info@silentvision.org.

8. TRAINING AUDIT
AWS SageMaker (EC2 p3.2xlarge, V100 GPU). TensorFlow + Keras. Adam optimizer.
Batch shape: (-1, 64, 224, 224, 3) for RGB / 5-channel for full model.

9. INFERENCE AUDIT
- Input: 64-frame sliding window with 32-frame stride
- Output: probability vector [Bag, Clothes, Normal]
- Latency: 5-second window minimum before first prediction at 30fps
- GPU required for real-time (V100 used in paper)
- NOT viable for CPU-only edge deployment

10. PRODUCTION
Hardcoded Windows absolute paths (E:\FINAL_PROJECT_DATA\...). Hardcoded Gmail credentials.
No API or web interface. ~5-second inference latency. GPU required.
Moderate production readiness conceptually but very rough implementation.

12. REUSABLE
- Optical flow as additional input channels — highly valuable concept for EDGE_AI
- 64-frame sliding window with overlap — good temporal window design pattern
- 3-class (Bag/Clothes/Normal) concealment taxonomy — aligns with EDGE_AI's signal types
- Multi-window max-confidence aggregation for variable-length video

13. VERDICT
Best ML architecture of all 17 repos. Not deployable as-is (Windows paths, hardcoded credentials,
GPU-only, 5s latency). But the optical flow + 3D-CNN approach is the most advanced idea to borrow
for a future EDGE_AI concealment model.

### REPO 7 AUDIT — TeCSAR-UNCC/Shopformer (CVPR 2025)

1. REPO OVERVIEW
CVPR 2025 academic paper. Two-stage anomaly detection operating entirely on skeletal pose sequences
(no RGB video). Privacy-preserving approach using GCAE (Graph Convolutional Autoencoder) + Transformer.

2. STRUCTURE
  models/         GCAE tokenizer + Transformer encoder-decoder
  scripts/        main_to.py — training + evaluation
  config/         Hyperparameter YAML configs
  utils/          Metrics, pose preprocessing
  data/           Dataset setup docs
  environment.yml (Python 3.13.2, CUDA 11.8, torch==2.6.0+cu118)

4. MODEL AUDIT
  GCAE        | Graph Conv Autoencoder | 18-keypoint pose graph -> 144-dim latent (8ch x 18 kpts), 2 tokens/seq
  Transformer | Encoder-Decoder        | 12 heads, 4 layers, dropout 0.1 — reconstructs tokenized pose sequences
  Anomaly score | MSE reconstruction error | High error = anomalous (shoplifting)

Training: only on NORMAL behavior. Inference: anomaly = high reconstruction error from normal distribution.
No supervised labels needed for shoplifting.

Dataset: PoseLift (custom) — 24-frame segments, 12-frame stride, 18 keypoints, binary labels.

Hyperparameters: LR=5e-5, epochs=10, Adam, MSE loss.

Performance: AUC-ROC 69.15%, AUC-PR 44.49%, EER 0.38. Modest — not production-grade accuracy.

5. FEATURES
- Unsupervised anomaly detection (no labeled shoplifting data needed)
- Privacy-preserving (processes only skeleton JSON, not raw video)
- Conda environment fully specified
- Apache 2.0 license (open for commercial use)

6. LOGIC
Train on normal -> at inference, shoplifting = deviation from learned normality (MSE threshold).
No thresholds on body part positions — learns what "normal movement" looks like holistically.

12. REUSABLE
- Privacy-preserving approach is valuable for multi-jurisdiction deployments
- Unsupervised training eliminates need for labeled shoplifting footage
- GCN for pose-graph modeling is a strong architectural idea for V2+ EDGE_AI
- GCAE tokenizer concept could augment pose analysis in EDGE_AI

13. VERDICT
Academically strong (CVPR 2025), but 69% AUC-ROC is too weak for production primary detection.
The unsupervised training approach removes the dataset labeling bottleneck.
Best use: supplemental anomaly signal alongside EDGE_AI's FSM.

### 5.2 Input Pipeline

VideoFileSource (src/video/sources.py:13):
- Loops video file via OpenCV VideoCapture
- Uses CAP_PROP_POS_MSEC for precise timestamp reconstruction — preserves video PTS so temporal drift
  does not accumulate
- Restarts from beginning on loop (configurable with loop=False)

RTSPSource (src/video/sources.py:42):
- Primary: cv2.CAP_FFMPEG with rtsp_transport;tcp, stimeout, fflags;nobuffer, flags;low_delay
- Fallback: generic cv2.VideoCapture(url) if CAP_FFMPEG fails
- Exponential backoff on reconnect (base 1s -> max 30s)
- Read failure tolerance: 3 consecutive failures before reconnect
- Stop event threading for clean shutdown

FrameSampler (src/video/frames.py):
- Rate controls frames to frame_fps (default 12fps)
- Drops frames when lagging (configurable)

### 5.3 Detector/Tracker Logic

YOLODetector (src/vision/detector.py):
- Single YOLO model call -> splits into persons (class 0) + items (RETAIL_COCO_CLASSES: 40 retail-relevant classes)
- Device autodetect: CUDA -> MPS -> CPU
- Two modes: detect_all() (persons + items) and track_persons() (ByteTrack, class 0 only)
- Cost: zero extra inference for item detection (same model call)

PersonTracker (src/vision/tracker.py):
- Primary: detector.track_persons() -> ByteTrack via ultralytics
- Fallback: _IOUFallbackTracker — IOU matching with 2-second track age
- TrackMemory: centroid history deque (maxlen=60), velocity calculation
- Velocity: Euclidean centroid displacement / dt (pixels/second)
- Automatic ByteTrack disable on exception

PoseEstimator (src/vision/pose.py):
- Batch inference: crops all tracked persons -> batch predict (single model call)
- COCO keypoints: indices 9/10 (wrists), 11/12 (hips)
- Derives: left_wrist, right_wrist, hip_center, hand_to_hip distance, hand_speed
- Confidence threshold: 0.20 (low, intentional — captures partial poses)
- Tracks hand position history per track_id for speed calculation

### 5.4 TheftRiskFSM — The Core Logic Engine (src/rules/theft_fsm.py)

States:
  BROWSING -> NEAR_SHELF -> SHELF_INTERACTION -> PICK_SUSPECTED
           -> POSSIBLE_CONCEALMENT -> CONFIRMED_CONCEALMENT
           -> HIGH_RISK_EXIT -> CLEARED

Signal weights:
  SHELF_INTERACTION:           2.0
  VISUAL_PICK_CONFIRMED:       4.0   (item disappeared near wrist)
  HAND_TO_POCKET:              4.0
  HAND_TO_BAG:                 3.0
  HAND_TO_PANTS:               4.5   (highest — trouser pocket is most intentional)
  HAND_TO_SHIRT:               3.5
  HAND_TO_HOODIE:              4.0
  EXIT_AFTER_CONCEALMENT:      6.0   (final commitment signal)
  REPEATED_SHELF_INTERACTION:  2.0
  NONSCAN_BAGGING:             5.0
  SPEED_SPIKE:                 1.5

Risk decay: 1.8 pts/second (time-based, not frame-count — stable across variable FPS inputs)

Pick detection (dual path):
  1. Visual: item track disappeared within 120px of wrist -> VISUAL_PICK_CONFIRMED (+4.0)
  2. Pose heuristic: wrist sustained inside shelf polygon for >=4 frames -> SHELF_INTERACTION (+2.0)

Concealment detection (multi-signal):
  1. Hand-to-hip distance <= 110px within conceal_window_sec after pick
  2. Wrist in bagging zone polygon (independent check)
  3. Classification via detect_conceal_type_from_pose():
     - BAG: wrist in bagging zone polygon
     - PANTS: wrist_y > hip_y + 15
     - HOODIE: wrist_y < shoulder_y - 20
     - SHIRT: wrist_y < mid_y
     - POCKET: default (hip level)

Exit detection: centroid crossed into exit_zone polygon (was outside before) + last_conceal_ts not None
  -> EXIT_AFTER_CONCEALMENT (+6.0)

Speed spike: velocity > 140 px/s while in exit zone -> SPEED_SPIKE (+1.5)

Event firing: risk_score >= risk_threshold (default 8.0) AND cooldown_ok (20s between events)

### 5.5 Per-Customer Basket State (src/rules/theft_state_machine.py)

Tracks per-customer item sets: items_in_hand, items_concealed, items_on_counter, items_returned

Signal-driven transitions and risk deltas:
  PICK                -> items_in_hand.add(item_id)              +1.0
  PUT_BACK            -> remove from all sets + returned          -5.0
  CONCEAL_*           -> in_hand -> concealed                    +15.0
  ON_COUNTER          -> in_hand -> on_counter                   -0.5
  RAPID_MULTI_PICK    ->                                          +2.0
  LOST_UNCERTAIN      ->                                          +3.0
  COUNTER_MISMATCH    ->                                         +20.0
  COUNTER_RECONCILED  ->                                         -10.0
  EXIT (with concealed) ->                                       +30.0
  EXIT (clean)        ->                                          0.0

### 5.6 RiskEngine (src/rules/risk.py)

Separate from FSM — operates on customer-level global IDs.
Decay: 2.0 pts per 30 seconds (slower decay for customer-level risk).
Clamps to [0, 100].
Risk bands: GREEN (<25), YELLOW (25-50), RED (>=50).
History: list of (ts, delta, reason) tuples for audit trail.

### 5.7 GlobalIdentityResolver (src/fusion/global_identity.py)

Composite matching score:
  0.55 * cosine_similarity(embedding)
+ 0.20 * time_score(last_seen, window=12s)
+ 0.15 * camera_adjacency_score(from_cam, to_cam)
+ 0.10 * height_similarity(height_px)

Match threshold: 0.72 (cosine).
Adjacency graph from cameras.sample.json (6-camera layout: entry, 4 shelf aisles, counter).
Thread-safe: uses threading.Lock() for all state mutations.

### 5.12 STRENGTHS OF EDGE_AI

1.  Architecture is genuinely production-grade for an MVP: temporal FSM, zone logic, multi-camera,
    ReID, evidence clips, API, UI, database
2.  Temporal decay is TIME-BASED (not frame-count-based) — correct and stable across variable FPS
3.  Dual-path pick detection: visual (item disappearance) + pose heuristic fallback — reduces missed detections
4.  Concealment type classification from COCO skeleton geometry (5 types: PANTS/SHIRT/HOODIE/POCKET/BAG)
    with different signal weights
5.  ByteTrack + IOU fallback — graceful degradation when ByteTrack fails
6.  Batch pose inference across all tracked persons — efficient (single model call per frame)
7.  Single YOLO call -> persons + items — zero extra inference cost for item detection
8.  RTSPSource with exponential backoff — production-grade reconnect logic
9.  PTS-based timestamps from video files — avoids timestamp drift in file replay
10. Thread-safe GlobalIdentityResolver with composite matching score (4-factor formula)
11. Background DB writer — non-blocking inference loop
12. SQLite corruption recovery — production stability consideration
13. Per-customer basket state tracking — item-level accounting (not just person-level)
14. Pydantic settings with validation — all thresholds configurable and validated
15. Evidence clip generation with pre/post event frames

### 5.13 WEAKNESSES OF EDGE_AI

1.  No training pipeline — zero support for training or fine-tuning models; model selection locked to YOLOv8n
2.  No benchmark/evaluation tooling — cannot compare models objectively on same test footage
3.  No model plugin system — swapping detector or pose model requires code changes
4.  No action/activity classifier — relies entirely on pose heuristics; no temporal video classifier
    (3D-CNN, LRCN, VideoMAE)
5.  Optical flow not used — the strongest concealment signal from academic research is missing
6.  No temporal classifier for shoplifting sequences — the FSM is expert-designed, not learned from data
7.  POS integration is a stub — src/pos/connector.py exists but not implemented
8.  Reconciliation logic is incomplete — routes_reconcile.py and reconciliation.py exist but POS data
    flow is absent
9.  ReID embedding quality unknown — src/vision/reid.py uses undocumented backbone (no OSNet/ResNet specs)
10. No hand/finger detection — wrist proxy is coarse; 120px pick radius may be too broad
11. Single mode threshold — risk_threshold=8.0 is a single global number; no adaptive or per-camera thresholds
12. No false positive rate measurement — no ground-truth evaluation pipeline exists
13. Item detection relies on COCO classes — custom retail items cannot be detected without custom training
14. No collective theft logic — only tracks individual persons; no cross-person coordination detection
15. No anomaly detection — all logic is explicit signals; no learned deviation from normal baseline
16. WebRTC implementation uncertainty — endpoint exists but real-world reliability not documented
17. UI is Streamlit — limited for production operational dashboards; no role-based access
18. Alert system is webhook-only — no email, SMS, WhatsApp, Telegram, Slack integration
19. No data labeling pipeline — no tooling to label events as true/false positives
20. No self-calibration — zone polygons must be manually configured; no automatic geometry estimation

### 5.14 MISSING COMPONENTS

- Training pipeline (fine-tuning YOLO/pose on custom retail classes)

- Collective theft detection (multi-person coordination)

- Custom retail item classes

- Operational dashboard with real-time metrics
- Data labeling and annotation tooling
- Camera calibration / auto zone setup

## 9. FINAL CTO VERDICT

### On the External Repositories

Do not be fooled by the 17-repo list. Of 17 repos:
  - 2 are non-functional (Repos 16, 17)
  - 6 are student notebooks with no deployment path (Repos 2, 8, 9, 10, 14, and code-less Repo 15)
  - 3 are hackathon demos (Repos 3, 6, and partially 5)
  - 3 have real research value but no production implementation (Repos 4, 7, 12)
  - Only 2 are genuinely worth studying for production patterns (Repos 1 and 11)

The 5 most valuable external repos for EDGE_AI:

  1. Repo 4 (Gate_Flow_SlowFast) — Best ML architecture: optical flow + 3D-CNN, real supermarket data,
     85.77% F1. Borrow the concept; build a lighter version.

  2. Repo 3 (POS Correlation) — Most realistic theft scenario: unrecorded sales. The POS-video
     cross-reference concept is immediately relevant for gas stations / convenience stores.

  3. Repo 13 (YOLOv8-pose + XGBoost) — Most deployable ML addition: lightweight, fast, privacy-
     preserving, adds a learned concealment classifier alongside existing heuristics.

  4. Repo 11 (LRCN package) — Best packaged temporal classifier: motion differencing + LRCN,
     pip installable, both live and file modes, designed for edge viability.

  5. Repo 7 (Shopformer) — Most academically advanced: GCAE+Transformer on skeleton sequences,
     CVPR 2025, unsupervised training, Apache 2.0 license, privacy-preserving.

### On EDGE_AI (EdgeGuard)

EdgeGuard is the best-architected system in this entire review. It has temporal FSM logic that none of
the 17 external repos even attempt. Its zone system, multi-camera ReID, basket tracking, evidence clip
generation, and production-grade RTSP handling are all superior to anything in the external repos.

The critical gap is the detection brain. EdgeGuard is currently a sophisticated reasoning system with
a single, fixed model for perception (YOLOv8n). You cannot experiment with models, cannot train custom
detectors, cannot measure false positive rates objectively, and cannot add learned temporal classifiers.
The rules layer is expert-designed; the perception layer needs a learning component.

Three immediate priorities for EDGE_AI:

  1. Model plugin interface + evaluation harness — without this, you cannot improve what you cannot measure

  2. Multi-channel alerting — operators in the field need mobile push, not just webhook

  3. POS integration — for gas stations and convenience stores, unrecorded transactions are the primary
     theft vector; the stub needs to become real

One honest criticism: The risk_threshold of 8.0 and the single set of signal weights are currently
global constants with no ground-truth validation. You do not know the actual false positive rate in
the field. This must be measured before any other optimization — you may be missing real events or,
more likely, firing on innocent customers. The evaluation harness (ST2) should be the very first new
code written.

Bottom line: EdgeGuard has strong bones. The architecture is right. The gaps are the detection
intelligence layer (ML models, training, evaluation) and the operational integration layer
(POS, multi-channel alerts, production dashboard). Fix those in that order.
